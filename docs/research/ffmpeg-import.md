# Research 2.3 — импорт-пайплайн: ffmpeg и энкодер для GIF → mp4

Дата: 2026-07-11. Статус: решение принято, реализовано в `apps/ui/src-tauri/src/library.rs`.

## Контекст

Правила проекта: GIF никогда не хранится и не играется как .gif — конвертация в mp4
при импорте; ffmpeg — только LGPL-сборка. Проблема: **в LGPL-сборках ffmpeg нет
libx264** (он GPL), а «параметры CRF 18–20» из плана — это x264-терминология.

## Что есть в запиненной LGPL-сборке (проверено `ffmpeg -encoders`, 2026-07-11)

- H.264: `h264_mf` (MediaFoundation), `h264_nvenc`, `h264_amf`, `h264_qsv`,
  `h264_d3d12va` — все аппаратные или системные, LGPL-совместимые;
- VP9: `libvpx-vp9` (BSD, software);
- AV1: `libsvtav1`, `libaom-av1` (BSD, software) + аппаратные варианты.

## Решение

1. **Основной энкодер — `h264_mf`** (Windows MediaFoundation):
   - есть на любой Windows 10+, включая машины без дискретного GPU
     (Microsoft поставляет software-фолбэк энкодера в составе ОС);
   - H.264 в mp4 — максимально совместимый выход (hwdec d3d11va при воспроизведении);
   - режим качества: `-rate_control quality -quality 70` (эмпирически: GIF 1.38 МБ,
     480x270\@15fps → mp4 1.05 МБ, воспроизведение позади иконок с hwdec подтверждено).
2. **Фолбэк — `libvpx-vp9`** (`-crf 32 -b:v 0`, тоже в контейнере mp4): на случай
   отказа MediaFoundation (например, урезанные N-редакции Windows без Media Feature Pack).
3. Общие параметры (из правил проекта): `scale=trunc(iw/2)*2:trunc(ih/2)*2`
   (чётные размеры), `-pix_fmt yuv420p`, `-movflags +faststart`, `-an`.

ffmpeg вызывается **отдельным процессом** (`CREATE_NO_WINDOW`), т.е. лицензионно
это использование, а не линковка; сборка тем не менее LGPL — консистентно с
правилами репозитория. Пин и хэш — в docs/third-party.md, загрузка —
`scripts/fetch-ffmpeg.ps1`.

## Превью

Один jpg-кадр `scale=320:-2 -q:v 4`; для видео берётся кадр на ~1s с повтором с 0s
для клипов короче секунды. Превью отдаётся в UI как base64 через invoke — asset
protocol не открывается (меньше поверхностей доступа).
