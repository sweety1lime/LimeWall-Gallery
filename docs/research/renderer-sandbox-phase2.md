# Песочница рендерера — Фаза 2: дизайн и план спайка

Статус: **дизайн, кода нет**. Продолжение [renderer-sandbox.md](renderer-sandbox.md).
Меняет зафиксированную архитектуру → требует согласования с владельцем (CLAUDE.md).

## Цель и не-цель

- **Цель:** изолировать **нативный декодер (libmpv/ffmpeg)** — самый ценный
  таргет: C-код парсит недоверенное видео, эксплойт → RCE. Фаза 1 (митигации+CFG)
  уже снижает эксплуатируемость; Фаза 2 добавляет **контейнмент** на случай пробоя.
- **Не-цель:** web/3D — их уже изолирует Chromium-песочница WebView2; не трогаем.

## Ключевое ограничение (из Фазы 1)

Декодер, запущенный в AppContainer/Low IL, **не может** ни `SetParent` в WorkerW,
ни рисовать через `--wid` в окно host'а (Medium IL) — UIPI блокирует
lower-IL-процесс, модифицирующий higher-IL-окно. **Вывод:** декодер не касается
окон вообще; рисует в **общую GPU-текстуру**, а окно держит и композитит host.

## Целевая архитектура (3 процесса)

```
ui (Tauri)  ──IPC──►  renderer = HOST (Medium IL)  ──ctrl-pipe──►  decoder (sandbox)
                       • окна-поверхности под WorkerW               • libmpv render API
                       • трей, watchdog, плейлисты, state           • декод 1 файла
                       • D3D11-композитор: рисует кадр decoder'а     • рендер в shared
                         на поверхность каждого монитора               D3D11-текстуру
                                                                     • без окон/сети/ФС
```

- **HOST** = нынешний `renderer` минус декодирование: владеет WS_CHILD-окнами под
  WorkerW, IPC с ui, трей, watchdog, плейлисты, персистенция. Для каждой активной
  медиа-поверхности спавнит decoder и **композитит** его кадры в окно (D3D11
  swapchain на HWND поверхности + отрисовка shared-текстуры, синхронизация keyed
  mutex).
- **DECODER** (новый бинарь, сильно ограничен): грузит libmpv в **render API**
  (`mpv_render_context_create`, off-screen FBO), декодирует один файл, пишет кадры
  в shared D3D11-текстуру. Никаких окон, сети, ФС кроме переданного хендла файла.
  Управляется host'ом по пайпу (load/pause/resume/volume/quality/screenshot).

## Общая поверхность (shared texture)

- Host создаёт `ID3D11Texture2D` с `D3D11_RESOURCE_MISC_SHARED_KEYEDMUTEX`,
  получает shared **NT-handle** (`IDXGIResource1::CreateSharedHandle`), передаёт
  его decoder'у при спавне (наследуемый handle).
- Decoder открывает текстуру (`OpenSharedResource1`), рендерит в неё кадр libmpv,
  берёт/отдаёт `IDXGIKeyedMutex`. Host под тем же mutex рисует её на swapchain.
- Вариант рендера libmpv: render API GL (ANGLE→D3D11) в GL-текстуру поверх
  D3D11-текстуры, **или** off-screen D3D11 gpu-context самого mpv. Какой путь
  дешевле — **вопрос Спайка 1**.

## Доступ к медиафайлу из песочницы

Не давать AppContainer доступ к ФС. Варианты (по предпочтению):
1. Host открывает файл на чтение и **передаёт наследуемый HANDLE**; decoder
   отдаёт libmpv через `fd://` / кастомный stream-cb. Ноль прав на ФС.
2. Иначе — добавить SID AppContainer в ACL **конкретного** файла (грубее).

## Профиль песочницы decoder'а

Т.к. у decoder'а нет JIT и он никого не спавнит, можно жёстко:
- **AppContainer** (свой профиль, без capability сети), либо (проще) **restricted
  token + Low IL**, если AppContainer+GPU окажется проблемным (Спайк 2).
- **ACG** (запрет динамического кода) — libmpv/ffmpeg без CPU-JIT, должно пройти.
- **Запрет дочерних процессов**, строгий токен, рабочая папка пустая.
- Наследуются только: shared-texture handle, ctrl-pipe, media handle.

## Бонус: устойчивость

Краш decoder'а (в т.ч. от эксплойта) больше **не роняет демон** — host ловит
выход, перезапускает decoder, обои продолжаются. Сейчас краш libmpv валит весь
renderer. Это отдельный плюс к надёжности беты.

## Спайки (сначала — самый решающий, без трогания боевого renderer)

1. **Пайплайн (make-or-break).** Отдельный эксперимент: decoder-процесс (пока
   **без** песочницы) рендерит libmpv render API в shared D3D11-текстуру; host
   композитит её в тестовое окно. Замерить перф (1080p/4K, неск. мониторов) и
   idle (пауза → ~0% CPU/GPU). Если дорого/криво — Фаза 2 под вопросом.
2. **Песочница.** Тот же decoder в AppContainer (иначе restricted token+Low IL):
   подтвердить доступ к GPU, открытие shared-handle, чтение переданного media
   handle. Подтвердить, что `SetParent(WorkerW)` из этого токена действительно
   падает (обоснование архитектуры).
3. **Жизненный цикл.** Краш→рестарт decoder'а host'ом; пауза/резюм; сверить
   якоря приёмки (автопауза, ~0% на паузе, restore).

## Разбивка на задачи (после зелёного Спайка 1)

- `crates/compositor` (или в platform): D3D11 swapchain на HWND + отрисовка
  shared-текстуры + keyed mutex.
- `crates/decoder` (новый бинарь): libmpv render API → shared texture, ctrl-pipe.
- Протокол host↔decoder (переиспользовать стиль `crates/ipc`).
- Спавн decoder'а в песочнице (в `crates/platform`, Windows).
- Интеграция в daemon: на каждую медиа-поверхность — decoder + композитор;
  watchdog считает CPU дерева host+decoder; web-путь без изменений.
- Обновить docs, PLAN.md, критерии приёмки.

## Риски и запасной план

- **Перф shared-texture-композита** хуже `--wid` (лишний проход/копия). Митигация:
  keyed mutex + zero-copy где можно; на паузе кадры не гоняем.
- **AppContainer + GPU** может потребовать возни с capability/loopback. Фолбэк:
  **restricted token + Low IL** (слабее AppContainer, но всё равно ниже medium и
  без прав на ФС) — уже большой шаг.
- **Стоимость велика** (недели, много unsafe D3D11/mpv FFI, GPU-специфика). Для
  беты с **модерируемой медиа-галереей** предельная польза сверх Фазы 1 +
  WebView2-песочницы **невелика**.

## Честная рекомендация

Дизайн — **готов** (эта заметка). **Реализацию Фазы 2 разумно отложить** до
приёма UGC / роста, а сейчас — по желанию владельца — сделать **только Спайк 1**
(time-boxed), чтобы узнать реальную стоимость/перф пайплайна и зафиксировать
вывод. Полную реализацию не начинать без зелёного Спайка 1 и согласования.
