# Research 1.1 — интеграция libmpv: биндинг и LGPL-сборка

Дата: 2026-07-10. Статус: **решение утверждено владельцем**, реализовано в `crates/mpv`.

## Вопрос 1: готовый Rust-биндинг или свой FFI

Кандидаты и их лицензии:

| Вариант | Лицензия | Вывод |
|---|---|---|
| `libmpv2` 6.0 (форк libmpv-rs, активен, май 2026) | **LGPL-2.1** | отклонён |
| `libmpv-rs` (оригинал) | **LGPL-2.1** | отклонён |
| собственный тонкий FFI | наш код | **выбран** |

Причина отклонения: Rust-крейты линкуются в бинарник **статически**. LGPL при
статической линковке требует обеспечить пользователю возможность перелинковки
(публикация объектных файлов или открытие кода) — несовместимо с планом
частично платного продукта. Динамическая линковка, которая решает вопрос для
самой libmpv-2.dll, к Rust-крейту неприменима.

Решение: `crates/mpv` — собственные объявления ~13 функций документированного
C client API (v2.x), написанные по официальной документации без копирования
заголовков mpv. Библиотека грузится в рантайме через `libloading` (ISC):

- LGPL-чистота: dll заменяема пользователем — обязательства LGPL выполнены;
- нет import-lib и link-time зависимости — renderer собирается без dll;
- внятная ошибка, если dll не найдена (подсказка про fetch-скрипт).

Цена решения: в фазе 5 (macOS) `--wid` недоступен — придётся добавить объявления
render API (~10 функций). Осознанно принято.

## Вопрос 2: откуда брать LGPL-сборку libmpv-2.dll

- Популярные сборки (shinchiro и зеркала) — **GPLv3**, не подходят.
- mpv собирается в LGPL-режиме опцией meson `-Dgpl=false`; ffmpeg внутри должен
  быть LGPL-конфигурации (без libx264 и пр. — нам не нужно, только декодирование).
- **Dev-этап (выбрано)**: артефакты `mpv-dev-lgpl-x86_64` из
  [zhongfly/mpv-winbuild](https://github.com/zhongfly/mpv-winbuild) — ежедневные
  GitHub Actions сборки с публичными логами; версия и SHA-256 запинены в
  `docs/third-party.md`, загрузка через `scripts/fetch-libmpv.ps1`.
- **Релиз**: собственная CI-сборка (та же конфигурация), т.к. мейнтейнер zhongfly
  прямо пишет, что не может дать юридических гарантий состава сборки.

## Заметки по интеграции (проверено вживую на Win10 22H2, 2026-07-10)

- Опции до `mpv_initialize`: `wid` (HWND поверхности), `hwdec=auto-safe`,
  `loop-file=inf`, `image-display-duration=inf` (картинки тем же пайплайном),
  `panscan=1.0` (заполнение без полос), `border=no` (артефакт 24H2),
  `stop-screensaver=no`, `input-default-bindings=no`, `osd-level=0`,
  `load-scripts=no`, `config=no`.
- `hwdec-current` устанавливается не сразу — опрашивать после старта декодирования
  (подтверждено: `d3d11va` для H.264 и HEVC 1080p, CPU 1.5–5% debug-сборки).
- `pause=true` реально останавливает декодирование: замерено 0% CPU.
- Статичная картинка (Motion JPEG 4K): 0% CPU после первого кадра.
- FSR-шейдер подключается в рантайме свойством `glsl-shaders` (путь с прямыми
  слэшами); включаем только когда исходник меньше монитора.
- Известное ограничение: если watchdog пересоздал окно поверхности (перезапуск
  explorer), HWND в `wid` устаревает — плеер нужно пересоздавать. Решается в
  фазе 2 событием от `WallpaperHost` (renderer-демон); для CLI фазы 1 — known issue.
