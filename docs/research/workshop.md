# Workshop — community gallery on GitHub (design)

Решение владельца (2026-07): **UGC сразу, хостинг GitHub, бесплатно.** Под эти
ограничения единственный безопасный для соло-издателя способ — **модерируемые
контрибуции через Pull Request**, а не мгновенная открытая заливка (последнее
требует сервера + детонации + модераторов, которых нет). Ниже — модель, где
UGC = контент сообщества с гейтом ревью, и она бесплатно ложится на GitHub.

Репозиторий один на весь проект (движок + галерея), контент галереи — в подпапке
`gallery/`, `.wpk`-бинарники — в GitHub Releases этого же репо. Контрибуции — PR,
затрагивающие `gallery/` (CI триггерится на эту папку).

Связано: [security-model.md](security-model.md) (контейнмент важнее детекции;
типы риска), формат `.wpk` (`crates/wpk`, манифест уже несёт author/license/version).

## Архитектура

Галерея живёт в **том же репозитории**, что и движок (владелец решил не разделять),
в подпапке `gallery/`:

- `gallery/catalog.json` — индекс, который тянет приложение (генерируется CI, подписан).
- `gallery/packs/<id>/manifest.json` + `preview.jpg` — метаданные и превью в гите (мелочь).
- `.wpk`-бинарники — в **GitHub Releases** (большие файлы, CDN `releases/download/…`,
  без квот Git LFS). Приложение читает `catalog.json` через Pages/raw, качает
  `.wpk` по release-URL.
- `revocation.json` — подписанный список отозванных id (kill-switch).

`catalog.json` (эскиз): `{ version, signature, packs: [{ id, name, author,
type, license, sha256, size, preview, download_url, tags, published_at }] }`.

## Поток загрузки = модерация (всё бесплатно на GitHub)

1. Контрибьютор форкает репо, кладёт пак, открывает PR.
2. **CI (GitHub Actions)** автоматически:
   - валидирует структуру `.wpk` (переиспользовать логику `crates/wpk`:
     traversal/симлинки/zip-бомбы/лимиты, схема манифеста);
   - **политика типов v1 — только `video`/`image`** (код `web`/`model3d` нельзя
     детонировать в CI → в UGC пока не пускаем; см. «Ограничения»);
   - капы разрешения/битрейта/длительности/размера; опц. AV-скан (free action);
   - **ре-энкод медиа через ffmpeg в CI** — нормализует и стирает вредоносную
     структуру файла (тот самый «серверный ре-энкод» из security-model, но за 0₽);
   - публикует результат в PR.
3. **Владелец мержит** — человеческий гейт модерации (глазами превью + вывод CI).
4. На мерж CI: заливает `.wpk` в Release, пересобирает и **подписывает**
   `catalog.json`, коммитит.

## Где безопасность (слои)

1. **Ограничение типов `video`/`image`** — самый мощный рычаг: инертное медиа,
   ноль исполнения кода. Снимает почти весь риск UGC.
2. **CI-валидация + ре-энкод** — структурные атаки и политика ловятся до ревью.
3. **PR-ревью человеком** — модерация; масштабируется временем владельца
   (ок на малом объёме; при росте — доверенные мейнтейнеры).
4. **Подпись пакетов/каталога (Ed25519) + revocation** — приватный ключ у 2fame
   (или в GitHub Secret для CI-подписи), публичный вшит в приложение; клиент
   **проверяет подпись перед установкой**; отзыв = удалить Release + внести id в
   `revocation.json` → клиент при старте сносит пак.
5. **Рантайм-контейнмент (уже есть)** — CSP-запрет сети, консент на код-обои,
   watchdog ресурсов — сеть на случай, если что-то просочится.
6. **Provenance/trust badge** — паки из галереи помечаются «проверено», сайдлоад —
   «не проверено».
7. **DMCA/копирайт** — у GitHub готовый takedown-процесс; в README галереи —
   политика приёма и удаления; клиентский revocation добивает. GitHub-хостинг тут
   прямой плюс.

## Отзыв (revocation) — реализовано в v1

Слой 4 выше описывает целевую модель (подпись Ed25519 + revocation). Подпись
**отложена** (см. security-model.md), но клиентский **kill-switch реализован**:

- `gallery/revocation.json` — `{ version, revoked: [{ id, sha256?, reason? }] }`.
  Инвариант (проверяет `validate.mjs`): отозванного пака **нет в каталоге** —
  сначала удаляем `gallery/packs/<id>/` и пересобираем каталог, потом вносим id.
- Клиент (`apps/ui/src-tauri/src/gallery.rs`):
  - `gallery_apply_revocations()` — при **старте** приложения: тянет список,
    удаляет из библиотеки элементы с отозванным id, останавливает их, если играют.
  - `gallery_fetch_catalog()` — при открытии «Каталога»: дополнительно **вырезает**
    отозванные паки из выдачи и делает ту же зачистку.
  - `gallery_download()` — повторно проверяет отзыв перед установкой (на случай
    устаревшего кеша каталога на фронте).
  - Совпадение по **id** (slug галереи = id элемента библиотеки для video/image;
    контент-хэши локальных импортов под это не попадают, ложных срабатываний нет);
    `sha256` дополнительно блокирует скачивание по хэшу `.wpk`.
- **Fail-open**: недоступность `revocation.json` (нет сети/404) → пустой список,
  галерея продолжает работать. Компромисс: атакующий, способный подавить этот файл,
  задерживает отзыв — это неотъемлемо для онлайн-отзыва. Без подписи целостность
  списка держится на HTTPS к нашему GitHub-домену и доступе к репозиторию; офлайн-
  подпись списка — следующий шаг усиления.

## Клиентская часть (код в этом репо)

- **UI: вкладка «Каталог»** — fetch `catalog.json` → грид с превью → «Скачать» →
  проверка подписи+sha256 → существующий `library_import`.
- **Rust-команды**: `gallery_fetch_catalog()` (HTTP GET + verify signature),
  `gallery_download(id) -> PathBuf` (скачать `.wpk`, сверить sha256+подпись, отдать
  импорту), `gallery_check_revocations()` при старте.
- **Зависимости**: лёгкий HTTP-клиент `ureq` (MIT) вместо тяжёлого reqwest;
  `ed25519-dalek` (BSD-3) для подписи. Обе — приемлемые лицензии (см. CLAUDE.md).
- Приложение впервые станет **сетевым** — только HTTPS к нашему GitHub-домену,
  без телеметрии. Каталог доступен офлайн-градусно: кешировать последний `catalog.json`.
- **Badge доверия** в библиотеке (`verified` из галереи vs `unverified` сайдлоад).

## Фазы

- **7a — Каталог + модерация (ядро)**: репо-галерея, CI-валидация+ре-энкод,
  формат `catalog.json`, подпись+revocation, клиентская вкладка «Каталог»
  (read/download/verify/import), badge доверия. **Только видео/картинки.**
- **7b — Позже**: in-app «Отправить пак» (гид/создание PR из приложения);
  `web`/`model3d` в UGC — только когда появится песочница/детонация;
  доверенные мейнтейнеры; расширенная модерация; (опц.) платные паки внешней
  ссылкой (Boosty/itch), разблокировка кодом — in-app платежи не делаем.

## Ограничения (честно)

- **Нет детонации песочницей** → `web`/`model3d` в UGC не принимаем в v1, даже
  через PR (ревью глазами не ловит обфусцированный JS; ре-энкодить код нельзя).
- **Модерация = время владельца** — не масштабируется бесконечно; на старте ок.
- **GitHub Releases** как файл-хранилище — есть soft-лимиты; следить за объёмом.
- Мгновенная открытая заливка (без PR-ревью) в этой модели **не поддерживается** —
  для неё нужен настоящий бэкенд с автоскан-конвейером и модераторами.
