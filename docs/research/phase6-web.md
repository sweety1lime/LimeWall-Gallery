# Research 6 — web/HTML и 3D-обои: подход и разбивка

Дата: 2026-07-11. Статус: **предложение, ждёт решения владельца перед кодом.**

## Вывод по осуществимости

Реализуемо на Windows и тестируемо на этой машине. Web-обои — это то же самое
окно-поверхность позади иконок, что у нас уже работает для mpv (`WorkerW` +
`SetParent`), только внутри не видеоплеер, а **WebView2** (движок Edge/Chromium).
Так делает Lively. 3D — частный случай web: three.js внутри того же webview
грузит glTF-модель. То есть `type: model3d` = `type: web` с забандленным
вьювером.

## Ключевое противоречие (нужно решение владельца)

Главный принцип проекта — **минимальная нагрузка**. Постоянно рисующий
Chromium позади рабочего стола — это ощутимый CPU/GPU 24/7, ровно то, чем
тяжелы конкуренты (Wallpaper Engine, Lively). Это прямой конфликт с нашим
позиционированием. Поэтому web-обои должны:

- жёстко подчиняться «вежливости»: при паузе — реальный `WebView2 TrySuspend`
  (заморозка процесса, не просто скрытие), проверить, что это даёт ~0% CPU;
- по умолчанию грузить только **локальный** HTML из `.wpk` (никаких
  произвольных URL в сеть — и нагрузка, и безопасность);
- честно предупреждать пользователя, что web/3D тяжелее видео.

Вопрос владельцу: **делаем ли Фазу 6 сейчас**, или её стоит отложить и вместо
неё заняться Фазой 5 (Linux/macOS — нужна не-Windows машина) либо полировкой
существующего? Web-обои расширяют возможности, но бьют по главному
преимуществу «лёгкости».

## Технические решения (если делаем)

- **Крейт**: `wry` (MIT/Apache) — обёртка над WebView2, которую использует Tauri;
  добавляем в `crates/renderer` напрямую (там Tauri нет). Runtime WebView2
  Evergreen есть в Win11 и большинстве Win10; если отсутствует — понятная ошибка
  (известная проблема Lively на Enterprise-редакциях без рантайма).
- **Ввод**: окно-ребёнок WorkerW не получает мышь/клавиатуру — для обоев это
  нормально (интерактивные web-обои — за рамками v1).
- **Безопасность**: только локальный `entry` из распакованного пакета,
  без node/произвольного сетевого доступа.

## Предлагаемая разбивка

- **6.1** `platform`: web-поверхность — WebView2 через `wry`, парентинг к
  WorkerW (переиспользуем discovery из `win32.rs`), пауза/резюм через
  `TrySuspend`/видимость. Проверить ~0% CPU в паузе (критерий «лёгкости»).
- **6.2** `renderer`/демон: тип сессии «web» рядом с mpv-сессиями; `play`
  для пакета `type: web` наводит webview на локальный `entry`. Вежливость и
  персист — как у видео.
- **6.3** `wpk`/библиотека: сейчас web/model3d отклоняются — принять их,
  распаковывать пакет целиком (не только `entry`), хранить папкой в библиотеке.
- **6.4** 3D: забандлить минимальный three.js-вьювер glTF; `type: model3d`
  грузит модель в него.
- **6.5** UI: карточки web/3D в библиотеке, применение как у видео; пометка
  «тяжелее видео».
- **Критерий приёмки**: HTML-обои играют позади иконок; при полноэкранной игре
  webview замораживается до ~0% CPU за ≤2 с; glTF-модель рендерится.
