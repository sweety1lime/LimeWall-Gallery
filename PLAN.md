# PLAN.md — роадмап LimeWall

Статус (2026-07-10):
- Фаза 0 — задачи 0.1–0.5 реализованы. Проверено на Windows 10 (22H2, один монитор): слой позади иконок, CPU ~0%, чистое снятие по Ctrl+C, устойчивость к смене обоев и F5. Для закрытия фазы осталось: Windows 11 (включая 24H2), мультимонитор, перезапуск explorer.exe.
- Фаза 1 — задачи 1.1–1.5 реализованы (свой FFI вместо LGPL-крейта — см. docs/research/libmpv.md; `--anime4k` использует официальный Mode B Fast; watchdog обновляет поверхность при смене разрешения/топологии дисплеев). Проверено вживую на Windows 10 22H2: H.264/HEVC 1080p через d3d11va (CPU 1.5–5% debug), pause → 0% CPU, картинка 4K → 0% CPU, FSR включается на 720p и отключается на нативном разрешении; Anime4K Mode B Fast загрузил все шесть шейдеров; `screenshot <path>` сохранил GPU-scaled кадры; цикл разрешения 1920x1080 → 1600x900 → 1920x1080 не прервал renderer, поверхность и кадры приняли правильные размеры. Предварительное сравнение Eco/FSR/Anime4K и topology evidence сохранены в `docs/comparisons/2026-07-10-win10-1080p.md`. Для закрытия фазы осталось: финальная визуальная/GPU-проверка на реальном 720p-аниме и 1440p/4K, физический hot-plug второго монитора.
- Фаза 2 — архитектура согласована (docs/research/phase2-architecture.md); срез 2.1a реализован в `crates/ipc`: protocol v1, валидация, ограниченный length-prefixed JSON framing, транспорт на `interprocess` (11 тестов). Срез 2.1b реализован и проверен вживую (2026-07-10): `renderer serve` + `renderer ctl ping|list-monitors|status|shutdown` через named pipe, чистый shutdown. Срез 2.1c реализован и проверен вживую (2026-07-10): демон владеет per-monitor сессиями, `ctl play|stop|pause|resume|volume|quality` (общий пайплайн вынесен в `renderer::playback`; соединения обрабатываются в отдельных тредах и пересылаются через mpsc на главный тред — зависший клиент блокирует только свой тред, что закрывает вопрос отсутствия таймаутов у named pipes; лимит 16 одновременных соединений). Живой прогон: FSR на 720p, hwdec d3d11va, pause → 0% CPU, переключение eco↔max на лету, замена контента с переиспользованием поверхности, MediaNotFound/no-session ошибки, чистый stop/shutdown с восстановлением стола. Каркас Tauri UI реализован (2026-07-10, остаток 2.1): `apps/ui` (Tauri 2 + vanilla TS + Vite, отдельный cargo-workspace через exclude), Rust-бэкенд находит демона или запускает его detached (`DETACHED_PROCESS|CREATE_NEW_PROCESS_GROUP`, поиск exe: LIMEWALL_RENDERER → рядом с ui.exe → target/), tauri-команды поверх ipc (connect/monitors/status/play/stop/pause/resume/volume/quality), панель: мониторы, выбор файла через нативный диалог, качество/громкость/Anime4K, сессии с автообновлением. Проверено: vite build, clippy, e2e-тест `ensure_daemon_starts_and_answers` (UI-бэкенд реально спавнит демона, ping, переиспользование, shutdown). Задачи 2.2/2.3 (библиотека + импорт) реализованы (2026-07-11): `%APPDATA%/LimeWall/library` с JSON-индексом (атомарная запись), content-hash id с дедупликацией, GIF → mp4 через LGPL ffmpeg (энкодер h264_mf + фолбэк libvpx-vp9 — решение в docs/research/ffmpeg-import.md, пин сборки в third-party.md), jpg-превью, сетка в UI с drag&drop, Import-диалогом, Apply/Remove. Проверено: 3 живых теста импорта (gif→mp4+превью+дедуп, png, отказ .txt), сконвертированный mp4 играет через демона с hwdec. Не проверено: живой запуск окна UI (нужен свободный рабочий стол). Задача 2.4 реализована (2026-07-11): демон сохраняет применённые обои в `%APPDATA%/LimeWall/wallpapers.json` (атомарная запись после каждой успешной playback-команды) и восстанавливает их при старте; монитор резолвится сначала по имени устройства (индексы плавают при смене топологии), потом по индексу; отсутствующие мониторы/файлы пропускаются, но остаются в конфиге; shutdown файл не трогает. Живой прогон: play(max, volume 20)+pause → рестарт демона → сессия вернулась paused с теми же параметрами и FSR → stop → конфиг пуст. Юнит-тесты резолва монитора и JSON round-trip. Задача 2.5 реализована (2026-07-11): трей принадлежит демону (`platform::tray`, скрытое окно на своём треде — не message-only, чтобы ловить TaskbarCreated и переживать перезапуск explorer; меню Pause all / Resume all / Open LimeWall / Quit, двойной клик = Open); автозапуск через HKCU Run (`platform::set_autostart`), IPC-команды Get/SetAutostart, `ctl autostart status|on|off`, галка «Start with Windows» в UI. Не проверено вживую: Open LimeWall из трея (откроет окно — ждёт свободного стола), брендовая иконка добавлена 2026-07-11 (в exe, окне и трее). Из фазы 2 остались приёмочные прогоны с участием владельца: визуальный запуск UI, полный сценарий с перезагрузкой ПК, чистая VM по README.

## Фаза 0 — «Цветной слой позади иконок» (убийца главного риска)

Цель: доказать, что мы умеем рисовать позади иконок рабочего стола на актуальных сборках Windows. Если это работает — работает всё приложение.

### Задачи

- **0.1 Исследование WorkerW.** Изучить актуальное состояние хака на Windows 10 и Windows 11 (включая изменения в 24H2): сообщение `0x052C` окну Progman, поиск WorkerW относительно `SHELLDLL_DefView`, известные обходы для новых сборок. Изучить подходы Lively и открытых примеров (идеи — да, код — нет, GPL). Результат — заметка `docs/research/workerw.md` с выводами и выбранной стратегией, показать владельцу до написания кода.
- **0.2 Каркас.** Cargo workspace, `crates/renderer`, `crates/platform`, `.gitignore`, README-заглушка.
- **0.3 Трейт `WallpaperHost`** в `crates/platform`: `enumerate_monitors() -> Vec<MonitorInfo>`, `create_surface(monitor_id) -> SurfaceHandle`, `destroy_surface(handle)`. MonitorInfo: id, имя, разрешение, координаты в виртуальном рабочем столе, масштаб (DPI).
- **0.4 Win32-бэкенд**: поиск Progman → `SendMessageTimeout(0x052C)` → поиск нужного WorkerW → создание собственного окна (crate `windows`) → `SetParent` → позиционирование строго под выбранный монитор (виртуальные координаты, DPI-aware) → заливка цветом.
- **0.5 CLI**: `renderer test-surface --monitor N --color "#RRGGBB"`. Корректное снятие по Ctrl+C: окно уничтожается, рабочий стол возвращается в исходное состояние.

### Критерии приёмки

- На Windows 10 и Windows 11 (в идеале включая 24H2) команда показывает цветной слой позади иконок; иконки видны и кликабельны.
- Мультимонитор: слой появляется только на указанном мониторе, координаты и DPI учтены.
- После завершения процесса рабочий стол полностью восстановлен: системные обои на месте, артефактов нет.
- CPU процесса в покое ~0%.
- Смена системных обоев / F5 на рабочем столе не ломает слой; если ломает — поведение зафиксировано в `docs/research/workerw.md` как known issue с планом обхода.

## Фаза 1 — Видео и картинки в слое

Цель: полноценное воспроизведение позади иконок с аппаратным декодированием и шейдерным апскейлом. Уже можно ставить обои с Майклом Джексоном.

### Задачи

- **1.1 Биндинг libmpv.** Выбрать поддерживаемый Rust-биндинг (кандидаты: `libmpv2`, `libmpv-rs` — проверить актуальность и лицензии). Подключить LGPL-сборку mpv (источник dll зафиксировать в `docs/third-party.md`).
- **1.2 Встраивание.** Передача HWND поверхности через `wid`. Опции: `hwdec=auto-safe`, `loop-file=inf`, `mute=yes` по умолчанию, заполнение экрана без чёрных полос (panscan/keepaspect), `stop-screensaver=no`.
- **1.3 Команды рендерера**: `play <path> --monitor N`, `stop`, `pause` / `resume`, `volume <0-100>`. На этом этапе достаточно CLI + простейшего управляющего канала (можно named pipe с JSON-командами — задел для `crates/ipc`).
- **1.4 Крючок апскейла.** Флаг `--quality eco|balanced|max`: eco = bilinear без шейдеров; balanced = lanczos (дефолт); max = GLSL-шейдеры FSR 1.0 (EASU+RCAS) из `assets/shaders`. Отдельный флаг `--anime4k` для аниме-профиля. Шейдеры автоматически не включаются, если исходник ≥ разрешения монитора.
- **1.5 Статичные изображения** тем же путём (mpv умеет показывать картинки) — один пайплайн на видео/гифки/фото.

### Критерии приёмки

- mp4 (H.264 и HEVC) 1080p играет в цикле позади иконок, без звука по умолчанию.
- Аппаратное декодирование активно (подтверждено статистикой mpv); CPU при 1080p-видео — единицы процентов на среднем железе.
- `pause` останавливает декодирование: CPU ~0%.
- 720p-видео на 1440p/4K мониторе: `--quality max` визуально резче, чем `eco`; сравнительные скриншоты сохранены в `docs/comparisons/`.
- Смена разрешения монитора / подключение второго монитора не роняет процесс (либо поведение зафиксировано как known issue).

## Фаза 2 — UI, библиотека и импорт

Цель: пользовательский сценарий «перетащил файл → применил на монитор» без командной строки.

### Задачи

- **2.1 Каркас Tauri 2** в `apps/ui` + минимальный IPC в `crates/ipc`: UI запускает/находит процесс renderer и шлёт JSON-команды (play/stop/pause/volume/quality per monitor).
- **2.2 Библиотека.** Папка данных (`%APPDATA%/LimeWall/library`), сетка с превью, drag&drop файлов в окно.
- **2.3 Импорт-пайплайн.** Видео копируется в библиотеку; **GIF → mp4** через ffmpeg (LGPL-сборка; параметры: `yuv420p`, чётные размеры, `movflags +faststart`, CRF ≈ 18–20); генерация превью (jpg-кадр). Опция «ужать исходники крупнее самого большого монитора» — по умолчанию ВЫКЛ (апскейлом занимается рендер, а вот 8K-исходник на 1080p-мониторе жать полезно).
- **2.4 Применение и состояние.** Кнопка «применить на монитор N»; выбор сохраняется в конфиг (json) и восстанавливается при старте рендерера.
- **2.5 Трей и автозапуск.** Закрытие окна UI не убивает рендерер; иконка в трее с меню (пауза всё / открыть UI / выход). Автозапуск Windows (ключ реестра Run), включается галкой в настройках.

### Критерии приёмки

- Сценарий целиком: перетащил `.gif` → в библиотеке появился элемент (уже mp4) с превью → клик «применить» → обои играют на выбранном мониторе.
- Закрытие окна UI: обои продолжают играть, процесс UI завершён, трей работает.
- Перезагрузка ПК: обои восстанавливаются без действий пользователя.
- Чистая Windows-VM: установка и запуск по README проходят без ручных правок.

## Фаза 3 — «Вежливость» и профили качества

Детализация (2026-07-11, см. docs/research/phase3-politeness.md):

- **3.1 Activity-watcher** в `crates/platform` (`platform::watcher`): полноэкранное приложение в фокусе (поллинг 500 мс, GetForegroundWindow + покрытие rcMonitor, исключая свой PID и Progman/WorkerW; пер-монитор по имени устройства), питание от батареи (GetSystemPowerStatus), блокировка сессии (WTSRegisterSessionNotification), выключение дисплея (GUID_CONSOLE_DISPLAY_STATE). События — change-only, наружу платформонейтральный enum.
- **3.2 Политика в демоне**: раздельные `user_paused` (персистится) и `effective_paused` (реально в mpv); reconcile-проход на каждое событие; resume пользователя не отменяет вежливость, пока условие активно; fullscreen действует только на свой монитор.
- **3.3 Политика батареи** `pause | eco | keep` (дефолт pause; eco = временный даунгрейд скейлеров/шейдеров с возвратом на AC): IPC Get/SetBatteryPolicy, `ctl battery status|pause|eco|keep`, селект в UI, персист в wallpapers.json.
- **3.4 Приёмка-якорь**: полноэкранная игра → нагрузка рендерера ~0% за ≤2 с.

Статус: реализовано и частично проверено вживую (2026-07-11) — на реальной полноэкранной игре: play при живой игре → auto-pause мгновенно, статус paused на +2.0 с, CPU 0% (якорь выполнен); resume во время игры остаётся paused; политика батареи персистится. Профили Эко/Баланс/Максимум per-monitor в UI были готовы в фазе 2. Осталось прогнать вживую при ручной приёмке: автовозобновление после закрытия игры, реальный переход на батарею (нужен ноутбук), блокировка сессии, выключение дисплея.

## Фаза 4 — Формат .wpk

`crates/wpk`: zip + manifest.json { id, type: video|image|web|model3d, entry, name, author, license, version, preview, options{} } — author/license/version обязательны с первой версии формата; валидация при импорте, экспорт любого элемента библиотеки в `.wpk`, импорт файла через UI и двойным кликом (ассоциация расширения). Шеринг v1 = обмен файлами.

Статус: реализовано (2026-07-11). `crates/wpk` (zip-крейт MIT): запись/чтение/валидация — обязательные поля, безопасные имена членов архива (без traversal/дисков), лимит манифеста 64 КБ, guard на размер файла; медиа кладётся в архив без сжатия (уже сжато кодеком). Библиотека UI: экспорт карточки в `.wpk` (кнопка + save-диалог), импорт `.wpk` через диалог/drag&drop/двойной клик; идентичность пакета сохраняется через `manifest.id` (санитизация id, фолбэк на content-hash), метаданные name/author/license переносятся; web/model3d честно отклоняются до фазы 6. Ассоциация `.wpk` регистрируется в HKCU при старте UI (без прав администратора), пакеты из argv импортируются при запуске. Проверено тестами: round-trip крейта (4), экспорт→импорт в свежую библиотеку с байт-в-байт совпадением mp4 и сохранением id/метаданных, регистрация/очистка ассоциации в реестре. Не проверено вживую: сам двойной клик (нужен рабочий стол); второй запуск UI передаёт .wpk работающему окну (single-instance, 2026-07-11).

## Фаза 5 — Linux, затем macOS (кратко)

Linux: бэкенд wlr-layer-shell (background) + X11 `_NET_WM_WINDOW_TYPE_DESKTOP`; GNOME Wayland — задокументировать как неподдерживаемый. macOS: NSWindow на desktop-уровне, все Spaces; libmpv через render API (wid недоступен). CI-матрица сборки под три ОС.

## Фаза 6 — Web/3D обои

Новый тип поверхности: webview-окно в слое обоев; three.js для glTF-моделей; `type: web | model3d` в манифесте. Нативный wgpu-рендер — только если упрёмся в производительность webview. Исследование и разбивка: docs/research/phase6-web.md.

Статус (2026-07-11):
- **6.1 готово** (коммит 09fb470): web-поверхность в `crates/platform` через WebView2 (`wry`). Две особенности Windows: WebView2 не создаётся на дочернем окне WorkerW — окно строится top-level, webview крепится, затем окно переподчиняется под WorkerW; и создание webview запускает вложенный message-pump, который блокируется внутри межпоточного SendMessageW — поэтому создание web-поверхности маршалится через PostMessage + канал. Watchdog пересобирает webview после перезапуска explorer. Проверено вживую: демо-страница играет позади иконок в отдельных процессах msedgewebview2 при 0% CPU рендерера, чистый выход. CLI `renderer test-web <file>`.
- **6.2 готово** (коммит 6a7daf2): демон различает mpv- и web-сессии по расширению (.html/.htm → web); «вежливость» для web = suspend webview (пауза за игрой/блокировкой/батареей); set_quality/set_volume на web корректно отклоняются; web-обои персистятся и восстанавливаются как видео. Проверено: play/status/stop/восстановление через `ctl`.
- **6.3 готово** (коммит 4d51a37): `wpk::extract_all` (распаковка всего пакета с лимитами на число файлов и размер против zip-бомб); библиотека хранит web-обои папкой `web-<id>/` с entry-HTML как `file`; импорт принимает `.wpk` типа web/model3d или голый `.html` (пакует его папку, с ограничением размера); экспорт перепаковывает всю папку; удаление сносит папку. Видео/картинки — без изменений. Тест: web-импорт→экспорт→реимпорт→удаление.
- **6.5 готово** (коммит 1519114): карточки web в UI (лаймовый плейсхолдер, подсказка «тяжелее видео»), импорт-диалог и drag&drop принимают `.html/.htm` и web-`.wpk`; применение web-карточки — как у любых обоев.
- **6.4 готово** (коммит 9126579): web-поверхность отдаёт папку пакета через внутренний протокол `wallpaper://` вместо `file://` — это разблокирует fetch/ES-модули/загрузку 3D-моделей (Chromium запрещает всё это из `file://`). Импорт `type: model3d` устанавливает вендоренный three.js-вьювер (three.module.min.js + GLTFLoader + BufferGeometryUtils, MIT) в папку пакета и генерирует `viewer.html`, который грузит модель, центрирует и авто-вращает. Проверено сквозняком: протокол отдал viewer.html, three.js-модули и `Box.glb` — модель загрузилась через fetch. Тест model3d-импорта в библиотеке.

**Фаза 6 завершена** (6.1–6.5). Осталась приёмка-якорь вживую: web-обои → suspend до ~0% CPU при полноэкранной игре в фокусе (механизм подключён, проверить с игрой).

## Фаза 7 — Workshop (галерея сообщества) — docs/research/workshop.md

Решение владельца (2026-07): UGC сразу, хостинг GitHub, бесплатно. Модель — **модерируемые контрибуции через PR** (не мгновенная открытая заливка): GitHub-репо-галерея, `catalog.json` (Pages/raw) + `.wpk` в Releases; паки попадают через PR с CI-проверкой и мержем владельца.

Безопасность встроена слоями: **тип-ограничение video/image в v1** (код web/3D не пускаем — нельзя детонировать); CI-валидация структуры (`crates/wpk`) + **ре-энкод медиа через ffmpeg в CI** (стирает эксплойты, бесплатно); PR-ревью = модерация; **подпись каталога/паков (Ed25519) + revocation-list** (клиент проверяет подпись перед установкой, отзыв = удалить Release + revocation → клиент сносит); рантайм-контейнмент уже есть (CSP/консент/watchdog); provenance-badge «проверено/не проверено»; DMCA через процесс GitHub.

- **7a (ядро)**: репо-галерея + CI (валидация+ре-энкод), формат `catalog.json`, подпись+revocation, клиентская вкладка «Каталог» (Rust `gallery_fetch/download` на `ureq` MIT + `ed25519-dalek` BSD → проверка подписи/sha256 → существующий `library_import`), badge доверия. Только видео/картинки. Приложение впервые сетевое (HTTPS к GitHub, без телеметрии, кеш каталога).
- **7b (позже)**: in-app «Отправить пак» (создание PR из приложения); web/3D в UGC — только после песочницы/детонации; доверенные мейнтейнеры. Монетизация: платные паки внешней ссылкой (Boosty/itch) + разблокировка кодом; ИИ-апскейл при импорте (Real-ESRGAN ncnn-vulkan). In-app платежи не делаем.

Мгновенная открытая заливка без PR-ревью потребовала бы полноценного бэкенда (автоскан-конвейер + модераторы) — вне текущего решения.

## Безопасность (сквозная) — docs/research/security-model.md

Защита пользователя от вредоносных обоев. Принцип: контент считается враждебным, контейнмент важнее детекции. Актуально уже сейчас (сайдлоад `.wpk`), критично для workshop.

Статус:
- **Готово ранее**: контейнер `.wpk` защищён (traversal/zip-бомбы/симлинки/лимиты, `crates/wpk`, тесты); web-поверхность без нативного моста; контент через `wallpaper://`, не `file://`.
- **Готово (эта итерация)**:
  - Сеть web-обоев заперта по умолчанию — CSP-заголовок `WALLPAPER_CSP` на каждом документе (`connect-src`/`script-src`/`frame-src` без remote-хоста → нет fetch/WS/remote-кода/iframe наружу), `nosniff`, navigation-handler держит top-frame на `wallpaper://`, new-window-handler запрещает `window.open`. Тест `served_assets_carry_the_containment_csp`.
  - Информированное согласие перед код-несущими обоями: web/3D `.wpk` и голый `.html` не устанавливаются молча (UI-инспекция `inspect_package` + предупреждение; двойной клик отдаёт UI событие `wpk-consent`, fail-safe). Медиа — без трения. Тест `only_code_bearing_packages_require_consent`.
  - Watchdog ресурсов: поток в демоне сэмплит CPU дерева процессов обоев (`platform::resources`, нативный Windows-замер), при устойчивом превышении (дефолт ≥25% машины ~20 с) латч-паузит виновника (web первым), латч снимается resume. Гистерезис против флаппинга. Тесты `breach_detector_*`, `resource_guard_*`, `cpu_percent_*`, `descendants_*`. Пороги требуют живой калибровки.
  - Причина паузы в UI: `SessionStatus.paused_reason` (`user|resources|game|battery|lock|display_off`, serde-default для version-skew); демон вычисляет по прецеденту `pause_reason`, индикатор «вежливости» показывает конкретику. Тест `session_status_reason_round_trips_and_defaults_to_none`.

Дальше (по приоритету, детали в заметке): песочница рендер-процесса (AppContainer × WorkerW — research); GPU-бюджет watchdog + пороги в настройках; badge «не проверено» в UI; подпись пакетов + revocation; серверный ре-энкод медиа; автоскан аплоада (валидация/AV/статанализ/детонация); уровни доверия + пост-модерация. Запуск workshop фазами: v1 — только видео/картинки; web/3D — позже за строгим ревью и песочницей.

## UX-улучшения (по болям пользователей конкурентов)

Основано на веб-исследовании жалоб к Wallpaper Engine / Lively (высокий CPU и непрозрачность нагрузки; «обои не показываются» на 24H2; отсутствие плейлистов).

Статус (готово, эта итерация):
- **Живой % CPU в UI** — watchdog шлёт `Message::CpuSample`, демон хранит последний сэмпл (freshness-guard `fresh_cpu`), `ResponseData::Status.stack_cpu_percent` (`serde(default)`), индикатор «вежливости» показывает «на рабочем столе · N% CPU»; `ctl status` печатает `stack CPU`; **tray-tooltip** «LimeWall — обоев: N · CPU: P%» (обновляется при изменении, `TrayGuard::set_tooltip` через общий `Arc<Mutex>` + NIM_MODIFY) — нагрузка видна без открытия окна. Тесты `fresh_cpu_*`, `status_extras_*`.
- **Плейлисты v1 (per-monitor, интервал + перемешивание)** — в демоне (работают при закрытом UI). ipc: `SetPlaylist/ClearPlaylist/PlaylistNext/GetPlaylist` + валидация (1..=512 items, 1..=1440 мин), `Status.playlists`. Чистая ротация `crates/renderer/src/playlist.rs` (Fisher–Yates + xorshift64, без зависимостей; тесты). Главный цикл демона на `recv_timeout` (без отдельного тикер-потока; `tick_playlists` после каждого пробуждения — иначе `CpuSample` каждые 4 с преемптит таймаут). Ротация откладывается пока обои на паузе (игра/блокировка/батарея); пропавший файл — скип+лог. Персистенция `PersistedState.playlists` (`serde(default)`, без бампа `STATE_VERSION`; старый демон восстановит текущий кадр как статичные обои). Ручной `Play`/`Stop` снимает плейлист. `ctl playlist set|clear|next|status`; трей «Next wallpaper». UI: «▶ Играть все» + поповер (интервал/перемешивание), полоса плейлиста в доке (следующие/остановить), герой следует ротации. Тесты `playlist::*`, `clamp_wakeup_*`, `persisted_playlist_round_trips`, `set_playlist_validation_*`.
- **Самодиагностика** — секция «Диагностика» в настройках: `run_diagnostics` проверяет демона (сырой ping, без спавна), renderer.exe, libmpv, ffmpeg, мониторы, **видимость иконок рабочего стола** (`platform::desktop_icons_visible` — 24H2-квирк), сессии, автозапуск, хвост журнала; кнопка «Скопировать отчёт». Тесты `tail_lines_*`, `read_log_*`.
- **Стартовые обои** — кнопка «Начать с примера» в пустом состоянии импортирует бандленный first-party web-сэмпл (`import_bundled_sample`, без консент-диалога как доверенный). Тест `web_folder_meta_*`.
- **Полировка** — слайдер громкости берёт значение из сессии при подключении (`adoptSessionControls` в `connect`).

Не проверено вживую (нужен демон+GUI): ротация плейлиста по времени, персист позиции при рестарте, отчёт диагностики на реальной машине, значение CPU vs Task Manager, tauri camelCase-арг `intervalMinutes`. Пороги CPU-watchdog — калибровать.

## Риски

- **WorkerW на новых сборках Windows** — главный технический риск; митигация: задача 0.1 и абстракция `WallpaperHost`.
- **GNOME Wayland** не даёт сторонним приложениям слой обоев — честно документируем ограничение.
- **Лицензии сборок mpv/ffmpeg** — использовать только LGPL-варианты, фиксировать источники (важно для будущей платности).
- **Чужой контент в галерее** (клипы, персонажи) — без модерации не запускать публичный каталог.
